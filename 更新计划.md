## 设计决策记录

### 触发时机
Event 在 `agent_end` 时通过 `EventManager.aggregateEvents()` 从 State 转移到 Memory SQLite。转移完成后，EventManager 立即回调 PersonalityModule，传递本次归档的 Events 列表。

### 人格更新输入
不再等待"昨日"，而是直接读取本次运行产生的 Events。每个 Event 包含：
- Plan 信息（任务类型、执行结果）
- Deviation 信息（偏差类型、严重程度）
- Attribution 信息（根因分析、调节措施）

### 人格更新输出
不再输出 Diary 文件，而是直接分析并选择性更新以下文件：
| 文件 | 维度 | 更新条件 |
|------|------|---------|
| `SOUL.md` | 自我 + 风格 + 信念 | 核心认知发生变化 |
| `IDENTITY.md` | 身份 | 角色/责任发生变化 |
| `skills/README.md` | 技能 | 新技能获取或旧技能强化 |
| `MEMORY.md` | 程序性记忆 | 新的 If-Then 规则形成 |

### 与 Agent 的职责边界
- **插件**：在 `agent_end` 时注入 development skill + Event 列表
- **Agent**：自行阅读 skill，判断每个 Event 对 6 个维度的影响类型（同化/顺应/无），决定更新哪些人格文件
- **插件**：接收 Agent 返回的更新内容，写入对应的人格文件

## 待讨论问题

1. **触发频率**：是否每次 `agent_end` 都触发人格更新？还是只在检测到 Deviation/Attribution 时才触发？
   - 建议：每次 agent_end 都触发，但传入的 Events 列表为空时跳过注入

2. **Event 数量**：单次运行可能产生多个 Event，development skill 是否设定了数量上限？
   - 建议：由 Agent 自行判断哪些 Event 值得分析，插件不干预

3. **异步执行**：人格更新是否应该异步执行，不阻塞 `agent_end` 的返回？
   - 建议：同步执行，`agent_end` 时顺便完成人格更新，不引入额外复杂度

4. **人格文件冲突**：如果 Agent 同时更新了多个文件，插件如何处理并发写入？
   - 建议：StateAdapter 使用 JSON 文件，天然串行；MemoryAdapter 使用 SQLite，事务保证
