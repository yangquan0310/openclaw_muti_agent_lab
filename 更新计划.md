## 设计决策记录

### 触发时机
Event 在 `agent_end` 时通过 `EventManager.aggregateEvents()` 从 State 转移到 Memory SQLite。PersonalityModule 在 `agent_end` Hook 中检测是否有新 Events 归档，若有则注入 development skill + Event 列表（通过 Hook 注入，非直接调用）。

### Event 生命周期（三阶段闭环）

```
第一阶段 planning    → Plan 创建（draft → pending_approval → active）
第二阶段 monitoring  → 循环执行（Session + Deviation + Attribution），Event 持续追加
第三阶段 regulation → 用户确认 → 归档（State → Memory）→ 人格更新
```

**一个 runId 一个 Event**，循环中持续追加，最终一次性归档（✅ 已确认）。

#### 创建时机：Plan 进入 draft 时（预创建 task JSON）
- **核心原则：任务开始前，预创建单个聚合 task JSON 到 State**
- 当 Plan 创建为 `draft` 时，插件创建 `task:{runId}`：
  ```json
  {
    "runId": "...",
    "status": "draft",
    "createdAt": "...",
    "updatedAt": "...",
    "plan": { "phases": [], "currentPhaseIndex": 0, "status": "draft" },
    "event": { "status": "draft", "deviations": [], "attributions": [], "planRevisions": [], "outcome": {} },
    "sessions": [],
    "tools": [],
    "activeSessionIds": []
  }
  ```
- **一个任务一个 JSON**，所有信息集中管理
- 后续循环中追加/更新 task JSON 的字段，无需检查对象是否存在
- Agent 也可读取 task JSON 了解完整进程

#### 更新时机：第二阶段循环中（追加模式，统一 task JSON）
- 每次出现 Deviation → 追加到 `task:{runId}.event.deviations[]`
- 每次完成 Attribution → 追加到 `task:{runId}.event.attributions[]`
- 每次 Plan 修订 → 追加到 `task:{runId}.event.planRevisions[]`，同时更新 `task:{runId}.plan`
- Session 状态变化 → 更新 `task:{runId}.sessions[]`
- 工具调用 → 追加到 `task:{runId}.tools[]`
- 第二阶段结束输出结果 → 写入 `task:{runId}.event.outcome`
- **State 只读写 `task:{runId}` 一个文件**（读取 JSON → 修改字段 → 写回）

#### 状态流转（含用户确认，统一 task JSON）
```
Plan.draft → task:{runId} 预创建（空壳）
     │
     ▼
用户审批 → Plan.active
     │
     ▼
第二阶段：循环执行
  ┌──────────────────────────────┐
  │ Session 执行 → LLM 输出      │
  │ 出现 Deviation？              │
  │   → 追加到 task.event.deviations[] │
  │   → 追加到 task.event.attributions[] │
  │   → 修订 task.plan            │
  │   → 更新 task:{runId}（State）│
  │   → 回到循环起点              │
  └──────────────────────────────┘
     │
     ▼
输出结果报告 → 更新 task.event.outcome
     │
     ▼
用户确认（✅ 使用状态机 pending_approval）
     │
  ┌──┴──┐
  ▼      ▼
未完成   完成
  │      │
  │      ▼
  │  task.status → completed
  │  进入第三阶段
  │    │
  │    ▼
  │  task:{runId} 整体归档到 Memory
  │  触发人格更新（✅ 立刻注入）
  │
  └──→ task.plan.status → draft → 重新审批（✅ 再走一遍审批）→ active → 回到第二阶段
```

**确认 ✅：Plugin asks, Agent decides, Plugin records**
- 插件职责：在 `agent_end` 时注入 development skill（含 Event 上下文）
- Agent 职责：自行阅读 skill、判断同化/顺应、决定是否更新人格文件
- 插件职责：接收 Agent 返回的更新内容，写入对应人格文件

### 人格更新输入
不再等待"昨日"，而是直接读取 `task:{runId}.event`，包含：
- `task.plan` 信息（任务类型、阶段执行结果）
- `task.event.deviations[]`（偏差记录：类型、严重程度、阶段）
- `task.event.attributions[]`（归因记录：根因、调节措施、执行结果）
- `task.event.planRevisions[]`（计划修订历史）
- `task.event.outcome`（最终结果报告）

**确认 ✅：每次第三阶段 agent_end 都触发，task.event 为空时跳过注入**

**确认 ✅：一个 runId 一个 task JSON，循环中持续追加，最终一次性归档**

**确认 ✅：State 实时覆盖写入，Memory 延迟到第三阶段一次性转移**

### Plan 状态机（含打回机制）

```
draft → pending_approval（Agent 汇报计划，等待用户审批）
  │         ↓ 用户不通过（打回）
  │      revising（Agent 在原 task 上修改 plan，不创建新任务）
  │         ↓ 修改完成
  │      pending_approval（重新提交审批）
  │         ↓ 用户通过
  │      active → 执行（第二阶段循环）
  │
  ↓
active → 执行（第二阶段循环）
  │
  ↓
pending_approval（第二阶段结束，输出结果报告，等待用户确认）
  │
  ├── 完成 → completed → 第三阶段归档 → Memory
  ├── 未完成（小修）→ revising → 重新审批 → active → 继续执行
  └── 用户提出新修改要求（重做/大改）→ revising → 重新制定 Plan → 重新审批 → active
```

**打回后状态**：`pending_approval` → `revising`（Agent 在原 `task:{runId}` 上修改或重写 plan，**不创建新任务**，保持同一个 runId）。修改完成后回到 `pending_approval` 重新审批。

**新增 `revising` 状态**：表示"计划正在修订中"，区别于 `draft`（初始草稿）。`revising` 支持 Plan 大幅重写，`task.planRevisions[]` 记录每次 Plan 版本历史。

**关键设计：不创建新任务**
- 无论是"未完成继续改"还是"用户提出新修改要求"，都回到 `revising`
- 保持同一个 `task:{runId}`，所有版本历史在 `task.planRevisions` 中追溯
- 只有当用户明确说"放弃这个任务，做一个全新的"时，才标记现有 task 为 `killed` 并创建新 `task:{newRunId}`

### 人格更新输出
不再输出 Diary 文件，而是直接分析并选择性更新以下文件。Agent 一次性返回 6 个维度的所有更新（方案 3，避免多次 LLM 调用）：

| 文件 | 维度 | 更新条件 |
|------|------|---------|
| `SOUL.md` | 自我 | 核心认知发生变化 |
| `SOUL.md` | 风格 | 表达/处理风格发生变化 |
| `SOUL.md` | 信念 | 价值观优先级发生变化 |
| `IDENTITY.md` | 身份（第 4 维） | 角色/责任发生变化 |
| `skills/README.md` | 技能（第 5 维） | 新技能获取或旧技能强化 |
| `MEMORY.md` | 程序性记忆 / If-Then 规则（第 6 维） | 新的条件-行动规则形成 |

### 工作记忆看板（Agent 自行维护，监控阶段自反性）

Agent 在自己的 `MEMORY.md` 中维护两个表格（插件不直接修改，通过 **monitoring skill** 自反性指导 Agent 自行更新）：

| 表格 | 作用 | 更新时机 |
|------|------|---------|
| **当前活跃任务看板** | 追踪当前活跃任务（runId、项目、描述、会话ID、状态） | **`pending_approval` → `active` 转换时**（执行前） |
| **活跃会话清单** | 追踪当前活跃会话（sessionId、类型/角色、分配任务、状态） | **`pending_approval` → `active` 转换时**（执行前） |

**更新方式**：用户同意执行计划后，monitoring skill 注入"自反性"指导，提醒 Agent 在执行前更新工作记忆看板。Agent 自行读取 `task:{runId}` 信息，更新 MEMORY.md 中的两个表格。

**清理由 Agent 自行判断**：任务完成归档后，Agent 自行标记 completed 或删除看板中的记录。

**确认 ✅：6 个维度独立（自我、风格、信念、身份、技能、程序性记忆）**

### 与 Agent 的职责边界（核心原则：Plugin asks, Agent decides, Plugin records）
- **插件**：在 `agent_end` 时读取 `task:{runId}.event`，注入 development skill + 完整 Event 信息作为 system context
- **Agent**：自行阅读 skill，判断 Event（含 deviations/attributions/outcome）对 6 个维度的影响，自行决定是否需要更新人格文件
- **插件**：接收 Agent 返回的更新内容（如果有），写入对应的人格文件；Agent 未返回更新时，插件不做任何操作
- **Agent 可将 task JSON 摘要写入 MEMORY.md**：development skill 指导 Agent 将任务中的关键 If-Then 规则从 `task.event` 提取并追加到 MEMORY.md，作为程序性记忆的追踪

> ⚠️ 插件**绝不**代替 Agent 判断同化或顺应，**绝不**主动修改人格文件。所有判断权在 Agent。
