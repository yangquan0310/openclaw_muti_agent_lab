## 一、更新概述

### 1.1 核心理念转变

| 维度 | 3.2.x 及之前 | 3.3.0（新方向） |
|------|-------------|----------------|
| **更新触发机制** | 时间驱动（每日定时任务） | 事件驱动（状态变化触发） |
| **自我更新方式** | 日记系统 + 每日总结 | 基于事件系统的即时响应 |
| **认知模式** | 被动回顾（事后总结） | 主动感知（实时响应） |
| **系统复杂度** | 高（日记存储、定时任务、回顾逻辑） | 低（事件监听、即时处理） |

### 1.2 为什么要取消日记系统和每日自我更新？

**日记系统的痛点：**
- ❌ **存储膨胀**：日记文件随时间线性增长，检索效率下降
- ❌ **信息滞后**：每日总结只能反映过去，无法指导当下
- ❌ **认知负担**：维护日记格式、回顾历史记录消耗 token
- ❌ **重复劳动**：很多日记内容与工作记忆、项目上下文重复

**每日自我更新的局限：**
- ❌ **固定节奏**：无法适应任务的实际节奏（有的任务需要即时更新，有的不需要）
- ❌ **批量处理**：一次性处理全天事件，容易遗漏关键细节
- ❌ **被动触发**：不管有没有重要事件，每天都执行，浪费资源

**事件驱动的优势：**
- ✅ **即时响应**：重要事件发生时立即处理，不等待
- ✅ **按需触发**：只有状态变化时才执行更新，节省资源
- ✅ **上下文保持**：事件处理天然携带上下文，无需额外检索
- ✅ **灵活扩展**：新类型的事件只需注册新处理器，无需修改核心逻辑

### 2.2 事件类型定义

#### 2.2.1 内置事件类型（核心）

| 事件类型 | 触发条件 | 优先级 | 处理目标 |
|---------|---------|--------|---------|
| `plan.created` | 新计划创建完成 | High | 工作记忆、计划看板 |
| `plan.approved` | 用户确认计划 | High | 执行状态、任务调度 |
| `plan.completed` | 计划全部阶段完成 | High | 归档、总结 |
| `phase.started` | 阶段开始执行 | Medium | 会话分配、工具准备 |
| `phase.completed` | 阶段完成 | Medium | 产出记录、状态更新 |
| `deviation.detected` | 检测到执行偏差 | Critical | 调节机制、告警 |
| `memory.updated` | 记忆内容变更 | Medium | 向量存储、持久化 |
| `session.ended` | 会话结束 | Low | 清理、归档 |

#### 2.2.2 扩展事件类型（可选）

| 事件类型 | 触发条件 | 用途 |
|---------|---------|------|
| `user.interrupt` | 用户中途修改需求 | 重新规划 |
| `tool.failed` | 工具调用失败 | 错误处理、降级 |
| `model.switched` | 模型切换 | 能力评估 |
| `config.changed` | 配置变更 | 热重载 |

### 2.3 事件处理器（Handler）设计

#### 2.3.1 认知更新处理器（Cognitive Update Handler）

```typescript
interface CognitiveUpdateHandler {
  // 处理计划相关事件
  onPlanCreated(event: PlanCreatedEvent): void;
  onPlanApproved(event: PlanApprovedEvent): void;
  onPlanCompleted(event: PlanCompletedEvent): void;
  
  // 处理偏差事件
  onDeviationDetected(event: DeviationEvent): void;
  
  // 更新目标：工作记忆、计划状态、认知上下文
}
```

**职责：**
- 维护当前活跃任务看板
- 更新计划执行状态
- 触发调节机制（当偏差发生时）

#### 2.3.2 记忆更新处理器（Memory Update Handler）

```typescript
interface MemoryUpdateHandler {
  // 处理记忆相关事件
  onMemoryUpdated(event: MemoryUpdatedEvent): void;
  onSessionEnded(event: SessionEndedEvent): void;
  
  // 更新目标：向量存储、持久化记忆
}
```

**职责：**
- 将重要事件向量化存入记忆
- 会话结束时归档关键信息
- 维护语义索引

#### 2.3.3 状态同步处理器（State Sync Handler）

```typescript
interface StateSyncHandler {
  // 处理状态变更事件
  onPhaseStarted(event: PhaseStartedEvent): void;
  onPhaseCompleted(event: PhaseCompletedEvent): void;
  onToolFailed(event: ToolFailedEvent): void;
  
  // 更新目标：状态存储、HEARTBEAT.md
}
```

**职责：**
- 同步 `plans.json` 状态
- 更新 `HEARTBEAT.md` 任务状态
- 维护会话清单

## 三、取消的具体内容

### 3.1 日记系统（Diary System）

#### 3.1.1 取消的组件

| 组件 | 当前位置 | 处理方式 |
|------|---------|---------|
| 日记文件存储 | `~/.openclaw/workspace/{agent}/diary/` | 归档到 `archive/diary/` |
| 日记模板 | `DiaryEntry` 类 | 删除 |
| 日记生成逻辑 | `generateDiaryEntry()` | 删除 |
| 日记检索接口 | `searchDiary()` | 删除 |
| 日记压缩机制 | `compressDiary()` | 删除 |

#### 3.1.2 替代方案

- **即时事件记录**：重要事件直接存入向量记忆，无需日记中转
- **会话级上下文**：每个会话的完整历史本身就是结构化记录
- **按需检索**：通过 `memory_search` 直接检索相关事件

### 3.2 每日自我更新（Daily Self-Update）

#### 3.2.1 取消的组件

| 组件 | 当前位置 | 处理方式 |
|------|---------|---------|
| 每日定时任务 | `HEARTBEAT.md` 中的每日任务 | 从 HEARTBEAT.md 移除 |
| 自我更新工作流 | Workflow 2（每日自我更新） | 删除 |
| 日记回顾逻辑 | `reviewDailyDiary()` | 删除 |
| 批量更新处理器 | `batchUpdateHandler()` | 删除 |

#### 3.2.2 替代方案

- **事件触发更新**：每个事件发生时即时处理，无需等待每日批次
- **状态实时同步**：`plans.json` 和 `HEARTBEAT.md` 随事件实时更新
- **增量持久化**：只有变化的部分才写入存储

## 六、风险与应对

### 6.1 风险识别

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|---------|
| 事件丢失 | 中 | 高 | 实现持久化事件队列，崩溃后可恢复 |
| 事件风暴 | 低 | 高 | 添加速率限制、批量处理机制 |
| 处理器故障 | 中 | 中 | 处理器隔离，单点故障不影响全局 |
| 迁移数据丢失 | 低 | 高 | 完整备份，迁移前验证 |
| 用户不适应 | 中 | 低 | 保留 3.2.x 分支，提供切换选项 |

### 6.2 回滚方案

如果 3.3.0 出现严重问题：
1. 停止事件系统
2. 恢复 `archive/diary-3.2.x/` 到 `diary/`
3. 恢复 `archive/heartbeat-3.2.x.md` 到 `HEARTBEAT.md`
4. 回滚代码到 3.2.x 分支
5. 重启服务

## 八、事件系统 API 预览

### 8.1 生产者 API

```typescript
// 触发事件
EventSystem.emit({
  type: 'plan.created',
  payload: {
    planId: 'uuid',
    goal: '任务目标',
    phases: [...]
  },
  timestamp: Date.now(),
  source: 'planning-skill'
});
```

### 8.2 消费者 API

```typescript
// 注册处理器
EventSystem.on('plan.created', async (event) => {
  // 更新工作记忆
  await workingMemory.updateActiveTask(event.payload);
  // 同步状态
  await stateSync.syncPlan(event.payload);
});
```

### 8.3 配置 API

```typescript
// 事件系统配置
interface EventSystemConfig {
  enabled: boolean;
  queueSize: number;        // 事件队列最大长度
  persistence: boolean;     // 是否持久化到磁盘
  rateLimit: number;        // 每秒最大事件数
  handlers: string[];       // 启用的处理器列表
}
```

## 九、验收标准

### 9.1 功能验收

- [ ] 事件系统能正确生产、路由、消费事件
