## 工作流

### 工作流1：标准调节流程

**步骤**：
1. **偏差根因分析**
   - 分析偏差的根本原因
   - 评估偏差对整体任务的影响

2. **影响范围评估**
   - 识别受影响的子任务
   - 评估是否需要调整最终交付物

3. **生成调节方案**
   - 方案A: [描述]
   - 方案B: [描述]
   - 方案C: [描述]

4. **选择最优方案**
   - 权衡成本、风险、收益
   - 确认方案可行性

5. **执行调节**
   - 更新任务看板状态（`paused` / `active`）
   - 通知相关子代理
   - 记录调节决策

6. **返回监控阶段**
   - 继续执行并跟踪

## 使用指南

### 输入参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `deviation_type` | string | ✅ | 偏差类型：progress / quality / resource / goal |
| `deviation_details` | object | ✅ | 偏差详情（来自 monitoring 阶段） |
| `current_plan` | object | ✅ | 当前执行计划 |

### 输出结果

| 输出项 | 格式 | 说明 |
|--------|------|------|
| `adjustment_log` | Markdown | 调节决策记录 |
| `updated_plan` | Markdown | 修正后的执行计划（如需要） |
| `notifications` | list | 需要发送给子代理的通知清单 |

### 常见偏差与应对策略

| 偏差类型 | 典型表现 | 调节策略 |
|----------|----------|----------|
| 进度落后 | 完成度 < 计划度 | 增加资源、并行执行、缩小范围 |
| 质量不符 | 输出不达标 | 调整方法、增加检查点、更换工具 |
| 资源不足 | 子代理/工具不可用 | 寻找替代资源、调整计划、请求支援 |
| 目标模糊 | 需求不明确 | 暂停任务、澄清需求、重新定义 |

### 状态更新格式

调节过程中，更新 `working_memory/subagent_tracker/SKILL.md`：

```markdown

| 项目 | 任务 | 负责子代理key | 状态 | 创建时间 | 最后更新 | 备注 |
|------|------|---------------|------|----------|----------|------|
| [项目] | [任务] | [key] | paused | [时间] | [现在] | 调节中: [原因] |

### 调节完成后
| 项目 | 任务 | 负责子代理key | 状态 | 创建时间 | 最后更新 | 备注 |
|------|------|---------------|------|----------|----------|------|
| [项目] | [任务] | [key] | active | [时间] | [现在] | 已调节: [方案] |
```
