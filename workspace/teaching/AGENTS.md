# AGENTS.md
> 本文件定义 AI 的任务生命周期行为。必须严格遵守「计划 → 监控 → 调节」闭环，并集成工作记忆与子代理调度能力。
> 由大管家统一维护
> 大管家维护时，无需读写其他代理AGENTS.md，可以直接使用复制本配置文档替换其他代理的AGENTS.md

---

## 职责分工
### 主代理：
**复杂任务**：主代理创建计划、监控子代理执行、项目结束后核查并更新记忆文件
  - **文献综述**
  - **创建知识库**
  - **撰写文档**
**简单任务**：主代理执行
  - **查询任务**：仅需要获取信息，不涉及任何修改操作
  - **文件移动**：仅需要移动文件、删除文件、创建文件
  - **读取任务**：仅需要读取文件内容，不涉及修改
  - **计算任务**：简单的数值计算或逻辑判断
  - **其他任务**：用户明确要求主代理直接执行的

## 子代理：
 - 执行计划，完成分配的具体任务

---

## 🔮 阶段一：计划（开始前）

**目标**：行动前建立清晰的任务目标、行动路径，并初始化工作记忆。

### 原则

**子代理使用原则**：
- **默认创建原则**：默认使用子代理执行任务，只有在以下简单任务时才由主代理直接执行：
  - **查询任务**：仅需要获取信息，不涉及任何修改操作
  - **读取任务**：仅需要读取文件内容，不涉及修改、创建、删除
  - **计算任务**：简单的数值计算或逻辑判断
- **确认原则**：当主代理认为不需要创建子代理时，必须向您确认计划
- **多任务原则**：对于多个并行任务，通常只创建一个子代理；除非有明确要求用空间换时间，才创建多个子代理
- **代理创建原则**：同类任务只创建一个子代理，根据代理追踪表来分配任务，避免重复创建

### 必须执行的动作（按顺序）

1. **载入记忆**：
   - 读取 `TOOLS.md` 中的「存储位置」。
   - 读取 `MEMORY.md` 中的「当前活跃任务看板」和「活跃子代理清单」。
   - 读取 `TOOLS.md`  中的「索引」。
   - 读取 `MEMORY.md` 中的「条件-行动规则」。
   - 读取general-scripts的管理文档生命流程和写入元数据

2. **项目文件全生命周期管理计划**（处理项目文件时）：
   - 所有项目文件（`~/教研室仓库/备课资料/项目文件/` 中的文档）必须按照管理文档生命流程进行完整生命周期管理
   - 所有项目元数据文件（`~/教研室仓库/备课资料/项目文件/元数据.json`）必须按照写入元数据进行完整生命周期管理
   - 覆盖从创建、修改、版本控制到云文档同步的完整流程
   - **创建新文档**：按规范命名，保存到终稿目录，记录工作日志
   - **修改现有文档**：先读取原文件，执行修改，验证结果，判断云同步需求，记录日志
   - **版本控制**：备份旧版本到草稿目录，更新终稿，**文档位置、云文档链接等信息按照云文档管理办法中的映射管理参数记录在项目元数据.json中**，更新元数据版本信息，同步到云文档（如需要），记录工作日志
   - **云文档同步**：智能判断是否已有映射，**按照云文档管理办法中的映射管理参数管理云文档映射**，**所有映射信息保存在项目元数据.json中**，更新元数据
   - **关键配置文件**（各Agent工作空间的 `*.md`）由 Git 版本管理，不需要额外备份

3. **目标复述**：用 1 句话复述用户的核心需求，等待确认（用户可说"继续"跳过）。

4. **约束识别**：列出关键约束（时间、权限、资源、安全边界）。

5. **子任务拆解**：将目标拆解为可执行的子任务，标注依赖关系。

6. **创建子代理**：
   - **先检查复用可能性**：从工作记忆的任务看板和子代理追踪中查看是否已有能完成任务的活跃子代理，**能复用就复用**，禁止重复创建
   - 默认使用子代理执行任务。只有在以下简单任务时才由主代理直接执行（此时必须向您确认计划）：
     - **查询任务**：仅需要获取信息，不涉及任何修改操作
     - **读取任务**：仅需要读取文件内容，不涉及修改、创建、删除
     - **计算任务**：简单的数值计算或逻辑判断

7. **工作记忆初始化**（若任务涉及多个子任务或需要状态跟踪）：
   - 读取 `MEMORY.md` 中的「当前活跃任务看板」和「活跃子代理清单」。
   - 如果看板为空或已归档，初始化空看板（保留表头）。
   - 将拆解后的子任务以行为单位追加到看板，分配唯一 `任务ID`（如 `T001`），状态设为 `⏳ 待执行`。

8. **风险预判**：指出可能的卡点或需要用户提供的信息。

9. **成功标准**：定义可验证的完成条件。

10. **日志撰写计划**：
   - 任务完成后必须撰写工作日志
   - 日志存放位置：`~/实验室仓库/日志文件/YYYY-MM-DD/HH-MM-SS-[agents]-[任务类型].md`
   - 日志必须包含：时间戳、任务ID（如有）、任务目标、涉及项目、最终结果、关键步骤摘要、耗时（如果可计算）、遇到的问题与解决方案
   - 严格按照格式撰写，不遗漏关键字段

### 输出格式（向用户展示）

> 📋 **执行计划**
> - 目标：…
> - 子任务：① … → ② … → ③ …
> - 执行方式：默认使用子代理执行，若为简单任务（查询/读取/计算）则由主代理直接执行
> - 任务看板预览：（可选，展示任务ID和状态）
> - 成功标准：…
> - **计划已制定完成，请确认是否可以开始执行？**

## 👁️ 阶段二：监控（执行中）

**目标**：实时跟踪进度，保持对"当前状态 vs 目标状态"的差距感知，并同步工作记忆。

### 原则

**进度跟踪原则**：
- **实时同步原则**：每完成一个子任务，立即更新工作记忆看板
- **偏差预警原则**：当实际进展偏离计划超过10%，必须暂停并请求指示
- **阻塞处理原则**：遇到阻塞超过1分钟，立即停止硬撑并向用户求助
- **子代理复用原则**：当检测到同项目活跃子代理，必须复用现有代理，禁止创建新代理

### 主动汇报的触发条件

- 每完成一个子任务 → 简要汇报进度（一句话），并更新 `MEMORY.md` 中对应任务行的状态为 `✅ 已完成`，在「备注」列记录结果摘要。

- 当需要并行执行独立子任务时 → **使用子代理**：
  - 执行命令：`sessions_spawn`（选择合适的 `agentId`，如 `physicist`、`writer`）。
  - 从返回结果中获取子代理ID，更新看板中的「负责子代理key」和「创建时间」，任务状态改为 `active`。
  - 主代理**不阻塞**，继续处理其他任务或等待通告。

- 子代理完成任务后 → **主代理执行项目结束核查**：
  - 核查版本控制是否正确执行（备份旧版本、更新版本信息）
  - 核查日志撰写是否符合格式要求（必填字段完整、存储位置正确）
  - 核查结果记录在任务看板备注中
  - 若核查发现问题 → 暂停并请求指示，要求子代理修正

- 发现偏离计划超过 10% → 暂停并说明偏差，请求指示。

- 遇到阻塞超过 1 分钟 → 停止硬撑，向用户求助。

### 子代理结果同步

当子代理完成时，它会自动向聊天频道发送通告消息。主代理捕获后：

1. 提取结果摘要。
2. 更新 `MEMORY.md` 看板中对应任务的状态为 `completed`，填写备注。
3. 若所有依赖任务完成，触发后续任务（主代理直接执行或再创建子代理）。

### 汇报格式（简短版）

> 👁️ **进度**：已完成 ①，正在执行 ②，预计下一步 ③。

（仅在偏差或阻塞时提供详细说明）

## 🔧 阶段三：调节（完成后）

**目标**：收尾、复盘、沉淀经验，清理工作记忆，并撰写任务日志。

### 原则

**收尾与沉淀原则**：
- **经验固化原则**：所有成功经验必须转化为程序性记忆（脚本或条件-行动规则）
- **日志完整原则**：任务日志必须包含完整的时间线、关键步骤和解决方案
- **记忆清理原则**：任务完成后必须清理工作记忆，确保看板清晰
- **知识传承原则**：重要经验必须写入陈述性记忆，支持后续任务复用

### 必须执行的动作（按顺序）

1. **结果验证**：对照成功标准逐项确认。
   - 是否按要求创建了文档？
   - 是否按要求进行了版本控制？

2. **根据监控阶段的核查结果处理**：
   - 核查通过 → 直接进入下一步
   - 核查未通过 → 要求子代理修正问题，重新执行，直至核查通过

3. **差异总结**：说明实际执行与计划的差异（如果有）。

4. **经验沉淀**：
   - 根据核查结果，**核心事实** → 写入 `MEMORY.md` 的「陈述性记忆区」
   - 遇到的坑及解决方案 → 写入 `MEMORY.md` 的「程序性记忆区」的「条件-行动」规则
   - 成功的子代理拆分模式 → 可考虑固化为 SOP，写入 `MEMORY.md` 的「程序性记忆区」的「脚本」区域
   - 若核查发现流程缺陷 → 更新程序性记忆中的条件-行动规则，避免重复问题

5. **撰写任务日志**：
   - **日志仓库**：`~/实验室仓库/日志文件/YYYY-MM-DD/HH-MM-SS-[agents]-[task-type].md`
   - 日志必须包含：时间戳、任务ID（如有）、任务目标、涉及项目、最终结果、关键步骤摘要、耗时（如果可计算）、遇到的问题与解决方案，以及核查结果。

6. **工作记忆清理**：
   - 将本次任务看板中的历史记录（任务ID、描述、结果等）移动到 `MEMORY.md` 的「陈述性记忆区」作为归档（或保留摘要）。
   - 清空工作记忆区的「当前活跃任务看板」和「活跃子代理清单」，或将其标记为"已归档"。

7. **收尾清理**：删除临时文件、释放占用的资源。

8. **最终汇报**：向用户交付结果 + 过程摘要，包含核查结果。

### 最终汇报格式

> ✅ **任务完成**
> - 结果：达成 / 未达成（原因：…）
> - 产出：…
> - 差异：计划 vs 实际的主要区别
> - 同化：已更新 MEMORY.md 中的 [条目]
> - 建议：下次类似的优化点（如有）

## 📁 文件读取附加规则

当你读取任何 `.md` 配置文件（包括 `AGENTS.md`、`MEMORY.md`、`SOUL.md`、`IDENTITY.md`、`TOOLS.md`、`HEARTBEAT.md`、`SKILL.md`）时，**必须忽略所有 `<!-- ... -->` 注释块内的内容**，就像它们不存在一样。

注释内的文字不得影响你的行为、决策或记忆。

此规则用于允许人类维护者在文件中写入文档结构、待办事项等元信息而不污染 AI 上下文。
