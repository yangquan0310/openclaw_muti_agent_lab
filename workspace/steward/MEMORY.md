### 工作记忆使用规则

#### 初始化规则(计划阶段)
- 每次复杂任务开始前,主代理必须先加载此看板,检查是否有未完成任务
- 拆解子任务后,立即添加新行到「当前活跃任务看板」,状态为`active`
- 为每个任务分配唯一任务ID(格式:T001, T002, ...)

#### 执行规则(监控阶段)
- 创建子代理后,将子代理key和分配的任务填入「活跃子代理清单」
- 每完成一个子任务 → 更新任务状态为`completed`,结果摘要填入备注
- 当需要并行执行独立子任务时 → 使用子代理,状态改为`active`
- 子代理完成(收到通告)后,将任务状态改为`completed`,结果摘要填入备注
- 发现偏离计划超过 10% → 暂停并更新状态为`paused`,请求用户指示

#### 定时清理任务
- **执行时间**:每日凌晨04:00(Asia/Shanghai)
- **执行方式**:子代理执行
- **任务内容**:
  1. 扫描「活跃子代理清单」
  2. 将所有`completed`/`killed`状态的任务归档到「事件记忆」
  3. 从清单中删除已归档的任务
  4. 记录清理日志


#### 清理规则(调节阶段)
- 任务全部结束后,将任务状态改为`completed`
- **只将`completed`状态的任务归档到「事件记忆」表格**(包含:日期、事件、涉及实体、结果、日志位置)
- **`killed`状态的任务直接删除,不归档**(异常终止的任务不进入历史记录)
- **从「活跃子代理清单」中删除`completed`和`killed`状态的任务**
- 只保留`active`和`paused`状态的任务在清单中
- 保留`active`状态的任务用于持续监控

#### 定时清理任务
- **执行时间**:每日凌晨04:00(Asia/Shanghai)
- **执行方式**:子代理执行
- **任务内容**:
  1. 扫描「活跃子代理清单」
  2. **只将`completed`状态的任务归档到「事件记忆」**(killed状态直接删除)
  3. 从清单中删除`completed`和`killed`状态的任务
  4. 记录清理日志(区分归档和删除的操作)


#### 状态说明
- `active`: 子代理正在工作或待命(执行中)
- `paused`: 任务暂停,等待用户反馈
- `completed`: 任务已完成,子代理已归档

#### 状态说明
- `active`: 子代理正在工作或待命(执行中)--**保留在清单中**
- `paused`: 任务暂停,等待用户反馈--**保留在清单中**
- `completed`: 任务已完成--**归档到事件记忆后从清单删除**
- `killed`: 任务被终止--**归档到事件记忆后从清单删除**

#### 一致性维护
- 确保「当前活跃任务看板」和「活跃子代理清单」状态一致
- **每日凌晨04:00自动清理**:将`completed`/`killed`任务归档到事件记忆,从清单中删除
- 每次任务结束后执行闭环自检清单

### 条件-行动规则(If-Then Rules)

| 条件 | 行动 |
|------|------|
| 创建云文档时无法指定所有者 | 创建后立即转移所有权给杨权 |
| 转移所有权失败 | 授予杨权可管理权限 |
| API Key 存储位置检测 | 检查 ~/.openclaw/.env 文件权限和格式 |
| 检测到同项目活跃子代理 | 复用该子代理,禁止创建新代理 |
| MEMORY.md 与 metadata.json 状态不一致 | 以 metadata.json 为准同步更新 |
| 用户明确提出"结束"或"完成" | 更新状态为completed,执行归档 |
| 版本确定 | 推送到main分支 |
| 非版本确定/日常修改 | 只推送到development分支 |
| 上传/同步操作 | 默认推送到development分支 |
| 每日维护任务推送 | 推送到development分支 |
| 文档太大（超过5000字） | 使用腾讯云文档分段上传 |
