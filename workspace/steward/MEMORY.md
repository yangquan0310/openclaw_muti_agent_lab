### 工作记忆使用规则

#### 初始化规则（计划阶段）
- 每次复杂任务开始前，主代理必须先加载此看板，检查是否有未完成任务
- 拆解子任务后，立即添加新行到「当前活跃任务看板」，状态为`active`
- 为每个任务分配唯一任务ID（格式：T001, T002, ...）

#### 执行规则（监控阶段）
- 创建子代理后，将子代理key和分配的任务填入「活跃子代理清单」
- 每完成一个子任务 → 更新任务状态为`completed`，结果摘要填入备注
- 当需要并行执行独立子任务时 → 使用子代理，状态改为`active`
- 子代理完成（收到通告）后，将任务状态改为`completed`，结果摘要填入备注
- 发现偏离计划超过 10% → 暂停并更新状态为`paused`，请求用户指示

#### 定时清理任务
- **执行时间**：每日凌晨04:00（Asia/Shanghai）
- **执行方式**：子代理执行
- **任务内容**：
  1. 扫描「活跃子代理清单」
  2. 将所有`completed`/`killed`状态的任务归档到「事件记忆」
  3. 从清单中删除已归档的任务
  4. 记录清理日志


#### 清理规则（调节阶段）
- 任务全部结束后，将任务状态改为`completed`
- **只将`completed`状态的任务归档到「事件记忆」表格**（包含：日期、事件、涉及实体、结果、日志位置）
- **`killed`状态的任务直接删除，不归档**（异常终止的任务不进入历史记录）
- **从「活跃子代理清单」中删除`completed`和`killed`状态的任务**
- 只保留`active`和`paused`状态的任务在清单中
- 保留`active`状态的任务用于持续监控

#### 定时清理任务
- **执行时间**：每日凌晨04:00（Asia/Shanghai）
- **执行方式**：子代理执行
- **任务内容**：
  1. 扫描「活跃子代理清单」
  2. **只将`completed`状态的任务归档到「事件记忆」**（killed状态直接删除）
  3. 从清单中删除`completed`和`killed`状态的任务
  4. 记录清理日志（区分归档和删除的操作）


#### 状态说明
- `active`: 子代理正在工作或待命（执行中）
- `paused`: 任务暂停，等待用户反馈
- `completed`: 任务已完成，子代理已归档

#### 状态说明
- `active`: 子代理正在工作或待命（执行中）——**保留在清单中**
- `paused`: 任务暂停，等待用户反馈——**保留在清单中**
- `completed`: 任务已完成——**归档到事件记忆后从清单删除**
- `killed`: 任务被终止——**归档到事件记忆后从清单删除**

#### 一致性维护
- 确保「当前活跃任务看板」和「活跃子代理清单」状态一致
- **每日凌晨04:00自动清理**：将`completed`/`killed`任务归档到事件记忆，从清单中删除
- 每次任务结束后执行闭环自检清单

#### S1 安全存储API密钥
```
触发条件: 需要安全存储新的API密钥

执行方式: 创建子代理执行

**子代理执行上下文：**
- 技能：文件操作（read/write/exec）
- 密钥：无需额外API密钥
- 使用方法：
  - read: 读取 ~/.openclaw/.env 文件
  - write: 写入API密钥到 ~/.openclaw/.env
  - exec: 执行 chmod 600 设置权限
- 任务要求：安全存储API密钥，设置文件权限为600
- 执行方案：检查文件→添加密钥→设置权限→验证

依赖: 无

输入: API密钥内容、服务名称（Zotero/Semantic Scholar/Baidu Scholar/其他）

步骤:
  1. 传递完整上下文给子代理
     - 提供API密钥内容和服务名称
     - 明确目标文件路径：~/.openclaw/.env
     - 规定输出格式：确认存储成功或错误信息

  2. 子代理执行任务
     - 确定API Key类型（Zotero/Semantic Scholar/Baidu Scholar/其他）
     - 检查 ~/.openclaw/.env 文件是否存在，不存在则创建
     - 在文件末尾添加新的API密钥变量，格式：SERVICE_NAME_API_KEY=your_api_key_here
     - 设置文件权限 chmod 600 ~/.openclaw/.env
     - 验证密钥格式和可读性

  3. 接收和验证子代理结果
     - 确认子代理返回存储成功确认
     - 验证文件权限是否正确设置

  4. 记录到工作日志

输出: API密钥已安全存储，文件权限设置为600
```

#### S2 维护配置文件
```
触发条件: 需要维护指定Agent的配置文件（AGENTS.md/SOUL.md/TOOLS.md/IDENTITY.md/MEMORY.md）

执行方式: 创建子代理执行

**子代理执行上下文：**
- 技能：文件操作（read/write/edit）
- 密钥：无需额外API密钥
- 使用方法：
  - read: 读取源配置文件和目标Agent配置文件
  - edit: 精确修改目标文件内容
  - write: 写入修改后的配置
- 任务要求：按照大管家配置结构维护其他Agent配置文件
- 执行方案：读取源文件→读取目标文件→按规范修改→验证→保存

依赖: 无

输入: 目标Agent名称、需要维护的配置文件类型、具体维护内容

步骤:
  1. 传递完整上下文给子代理
     - 提供目标Agent名称和配置文件类型
     - 明确维护内容（新增/修改/删除的具体部分）
     - 提供大管家参考配置的结构规范
     - 规定输出格式：修改摘要和文件路径

  2. 子代理执行任务
     - 确定目标Agent（数学家/物理学家/心理学家等）
     - 确定配置类型（SOUL.md/IDENTITY.md/MEMORY.md等）
     - 明确具体的维护内容（新增/修改/删除哪些部分）
     - 参考大管家配置结构：
       * 各个小节按照大管家在当前配置文件中**当前小节的批注**来维护其他代理配置
       * 对于共享结构（如AGENTS.md完整结构）：大管家更新后，其他代理直接整体替换
       * 对于独立内容（如陈述性记忆）：各个代理维护自己的内容，大管家仅维护格式（行间距、列表格式、表格对齐等）
       * 只修改指定内容，保留其他原有内容
     - 读取目标Agent现有配置文件
     - 根据维护内容进行修改，保持大管家结构
     - 写入目标Agent工作空间

  3. 接收和验证子代理结果
     - 确认子代理返回修改摘要
     - 验证文件完整性
     - 验证修改内容是否正确
     - 确认其他配置未受影响

  4. 记录到工作日志

输出: 配置文件更新完成，包含修改摘要和文件路径
```

### 条件-行动规则（If-Then Rules）

| 条件 | 行动 |
|------|------|
| 创建云文档时无法指定所有者 | 创建后立即转移所有权给杨权 |
| 转移所有权失败 | 授予杨权可管理权限 |
| API Key 存储位置检测 | 检查 ~/.openclaw/.env 文件权限和格式 |
| 检测到同项目活跃子代理 | 复用该子代理，禁止创建新代理 |
| MEMORY.md 与 metadata.json 状态不一致 | 以 metadata.json 为准同步更新 |
| 用户明确提出"结束"或"完成" | 更新状态为completed，执行归档 |
| 公告超过30天 | 自动归档到归档文件夹 |
