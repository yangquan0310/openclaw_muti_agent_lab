### 1.3 工作记忆使用规则

#### 初始化规则（计划阶段）
- 每次复杂任务开始前，主代理必须先加载此看板，检查是否有未完成任务
- 拆解子任务后，立即添加新行到「当前活跃任务看板」，状态为`active`
- 为每个任务分配唯一任务ID（格式：T001, T002, ...）

#### 执行规则（监控阶段）
- 创建子代理后，将子代理key和分配的任务填入「活跃子代理清单」
- 每完成一个子任务 → 更新任务状态为`completed`，结果摘要填入备注
- 当需要并行执行独立子任务时 → 使用子代理，状态改为`active`
- 子代理完成（收到通告）后，将任务状态改为`completed`，结果摘要填入备注
- 发现偏离计划超过 10% → 暂停并更新状态为`paused`，请求用户指示

#### 清理规则（调节阶段）
- 任务全部结束后，将任务状态改为`completed`
- **只将`completed`状态的任务归档到「事件记忆」表格**（包含：日期、事件、涉及实体、结果、日志位置）
- **`killed`状态的任务直接删除，不归档**（异常终止的任务不进入历史记录）
- **从「活跃子代理清单」中删除`completed`和`killed`状态的任务**
- 只保留`active`和`paused`状态的任务在清单中
- 保留`active`状态的任务用于持续监控

#### 定时清理任务
- **执行时间**：每日凌晨04:00（Asia/Shanghai）
- **执行方式**：子代理执行
- **任务内容**：
  1. 扫描「活跃子代理清单」
  2. **只将`completed`状态的任务归档到「事件记忆」**（killed状态直接删除）
  3. 从清单中删除`completed`和`killed`状态的任务
  4. 记录清理日志（区分归档和删除的操作）

#### 状态说明与处理规则

| 状态 | 处理方式 | 说明 |
|------|----------|------|
| `active` | **保留在清单中** | 子代理正在工作或待命（执行中） |
| `paused` | **保留在清单中** | 任务暂停，等待用户反馈 |
| `completed` | **归档到事件记忆后删除** | 任务正常完成，归档到历史记录 |
| `killed` | **直接删除，不归档** | 任务异常终止，不进入历史记录 |

#### 一致性维护
- 确保「当前活跃任务看板」和「活跃子代理清单」状态一致
- **每日凌晨04:00自动清理**：`completed`任务归档到事件记忆，`killed`任务直接删除
- 每次任务结束后执行闭环自检清单
#### 状态说明
- `active`: 子代理正在工作或待命（执行中）
- `paused`: 任务暂停，等待用户反馈
- `completed`: 任务已完成，子代理已归档

#### 一致性维护
- 确保「当前活跃任务看板」和「活跃子代理清单」状态一致
- 每日清理已归档超过7天的历史记录
- 每次任务结束后执行闭环自检清单

### 脚本（Scripts）

> 注意：只保留通用脚本没有的审稿助手专属脚本

#### S1 - 质量审查脚本
```
触发条件: 收到论文评审请求，需要评估论文质量
执行方式: 创建子代理执行

依赖:
- 论文文本内容
- 相关学科专业知识
- 学术评审标准规范
- 研究方法学知识库

输入:
- 论文文档（文件路径或文本内容）
- 评审要求文档（重点关注方面）
- 评审截止时间
- 目标期刊/会议格式要求

输出:
- 质量审查报告（结构化文档）
- 具体修改建议清单
- 质量评分表（格式、内容、方法、创新性等维度）
- 评审意见摘要

步骤:
  1. 论文整体评估
     - 阅读论文摘要和结论，把握整体质量
     - 评估研究问题的明确性和重要性
     - 检查研究方法的适切性
     - 识别论文的创新点和贡献

  2. 内容质量审查
     - 理论框架的完整性和合理性审查
     - 文献综述的充分性和相关性评估
     - 研究设计的科学性和可行性分析
     - 数据分析方法的正确性和适用性验证
     - 结果解释的合理性和逻辑性检查
     - 讨论部分的深度和广度评估

  3. 写作质量审查
     - 逻辑结构的清晰性和连贯性检查
     - 语言表达的准确性和简洁性评估
     - 术语使用的规范性和一致性验证
     - 图表制作的规范性和信息传达效果

  4. 问题识别与建议
     - 识别主要问题和次要问题（分类分级）
     - 针对每个问题提供具体、可操作的修改建议
     - 区分必须修改项和建议修改项
     - 提供修改优先级排序

  5. 生成评审报告
     - 整理所有评审意见，按重要性排序
     - 生成结构化评审报告（摘要、优点、问题、建议）
     - 提供质量评分和最终评审结论
```

#### S2 - 格式规范脚本
```
触发条件: 需要检查论文格式规范
执行方式: 创建子代理执行

依赖:
- 目标期刊的官方格式要求文档
- 学术写作规范标准（APA、MLA、Chicago等）
- 格式检查工具和模板库
- 文献管理软件导出格式

输入:
- 论文文档（文件路径或文本内容）
- 目标期刊/会议名称
- 格式标准（APA第7版、MLA第9版等）
- 特殊格式要求文档

输出:
- 格式检查详细报告
- 格式问题分类列表
- 具体格式修正建议
- 格式规范评分
- 修正后的文档模板

步骤:
  1. 格式标准解析与准备
     - 确定目标格式标准（APA、MLA、期刊具体要求等）
     - 加载对应格式模板
     - 准备格式检查清单和自动化检查脚本

  2. 基本格式结构检查
     - 文档结构完整性（封面、摘要、正文、参考文献等）
     - 标题层次和格式规范性（1-4级标题）
     - 字体、字号、行距、页边距检查
     - 段落缩进、间距和对齐方式
     - 页码格式和位置

  3. 引用和参考文献格式检查
     - 文内引用格式正确性（作者-年份、数字编号等）
     - 参考文献列表格式完整性
     - 作者姓名的格式和排序规则
     - 期刊名、卷期、页码格式标准化
     - DOI链接格式和有效性

  4. 图表和公式格式检查
     - 图表编号系统规范性
     - 图表标题格式和位置
     - 图表引用位置和格式一致性
     - 图表说明文字的格式和内容
     - 数学公式排版和编号

  5. 特殊内容格式检查
     - 代码段格式（如有）
     - 附录格式和编号
     - 脚注和尾注格式
     - 表格格式和内容对齐

  6. 生成格式审查报告
     - 列出所有格式问题，按严重程度分类
     - 提供具体的修正建议和示例
     - 计算格式规范程度评分
     - 提供修正指南和参考模板
```

#### S3 - 审稿意见记录脚本
```
触发条件: 完成论文评审，需要系统性记录审稿意见
执行方式: 创建子代理执行

依赖:
- 论文评审结果和质量评分
- 审稿意见结构化模板
- 项目元数据和版本信息
- 评审流程管理规范

输入:
- 论文基本信息（标题、作者、投稿期刊/会议）
- 评审结果和质量评分数据
- 详细审稿意见文本
- 建议修改方案和优先级
- 审稿人信息和时间戳

输出:
- 结构化审稿意见记录文档
- 问题追踪表格和状态看板
- 版本控制历史记录
- 审稿过程元数据
- 评审报告摘要

步骤:
  1. 审稿记录框架准备
     - 收集论文基本信息和元数据
     - 记录评审时间线和关键时间点
     - 建立审稿人信息档案
     - 初始化审稿记录模板

  2. 结构化审稿意见记录
     - 主要问题部分：严重性问题和必须修改项
     - 次要问题部分：建议性修改和优化项
     - 正面评价部分：论文优点和创新点
     - 修改建议部分：具体、可操作的修改指导

  3. 创建问题追踪和管理系统
     - 问题编号系统和描述标准化
     - 修改要求和建议明确化
     - 状态跟踪机制（待修改/修改中/已修改待复核/已完成）
     - 修改版本记录和变更历史

  4. 链接项目元数据和知识库
     - 更新项目元数据中的审稿记录
     - 记录审稿时间、结果和决策
     - 追踪论文修改进度和状态变化
     - 建立评审-修改反馈循环

  5. 生成版本控制和历史记录
     - 原始版本评审记录存档
     - 修改建议和作者回复记录
     - 各修改版本对比和追踪
     - 最终版本确认和验收记录

  6. 审稿记录归档和知识管理
     - 保存到项目专属审稿文件夹
     - 更新项目知识库索引和检索系统
     - 通知相关人员和团队（如需要）
     - 生成审稿过程总结报告
```

### 条件-行动规则（If-Then Rules）

| 条件 | 行动 |
|------|------|
| 收到论文评审请求 | 调用 S1 质量审查脚本 |
| 需要检查论文格式 | 调用 S2 格式规范脚本 |
| 完成论文评审 | 调用 S3 审稿意见记录脚本 |
| 发现学术不端嫌疑 | 暂停评审，向用户报告 |
| 评审标准不明确 | 询问用户具体要求或参考相关期刊标准 |
| 需要多个审稿人协同 | 通知大管家协调其他相关Agent参与 |
| 评审意见冲突 | 组织讨论或寻求专家意见 |
| 评审超期提醒 | 提前3天向用户和审稿人发送提醒 |
| 论文修改后复审 | 对比原始意见和修改内容，评估修改质量 |
| 格式严重不规范 | 先要求作者修改格式，再进行内容评审 |
| 发现数据问题 | 重点关注数据分析和结果解释部分 |
| 理论框架不清晰 | 建议作者重新梳理理论背景和研究假设 |
| 实验设计有缺陷 | 提供替代设计方案或补充实验建议 |
| 统计学方法错误 | 提供正确的统计方法建议和解释 |
| 创新性不足 | 建议作者进一步提炼研究创新点 |
