# v3.4.0 更新计划：任务创建延迟与 Agent 自主决策

## 问题诊断

当前 v3.3.0 实现中，`MetacognitionModule.onBeforePromptBuild()` 在每次 `before_prompt_build` 时**自动预创建** `task:{runId}.json`：

```javascript
// 当前逻辑（问题所在）
async onBeforePromptBuild(event, ctx) {
  let task = await this._getOrCreateTask(runId); // ← 无条件创建
  if (!task.plan || !task.plan.execution) {
    // 自动生成 plan 模板...
    await this._saveTask(task);
  }
}
```

这导致：
1. **每个 runId 都会创建一个 task JSON 文件**，即使是简单查询（"今天天气"、"翻译这句话"）
2. 插件替 Agent 做了**任务复杂度判断**（`_shouldUseMetacognition`），违反核心原则
3. 大量 `draft` 状态的 task 文件从未被使用，占用存储且需要后续清理

**核心矛盾**："Plugin asks, Agent decides, Plugin records" —— 但当前插件在 Agent 尚未决定前就**自动创建**了任务容器。

---

## 设计目标

- **插件不替 Agent 判断**任务是否需要 Plan
- **Agent 自主评估**用户内容的复杂度，决定是否创建 Plan
- **延迟创建 task JSON**：只在 Agent 明确决定需要 Plan 后才创建
- **简单任务零开销**：查询/翻译/计算等简单任务不创建任何 task 文件

## 新状态流转

```
用户提问
    │
    ▼
before_prompt_build
    │
    ├── task 不存在 ──────────────────────────────┐
    │                                              ▼
    │                                    注入 assessment skill
    │                                              │
    │                                    Agent 评估任务复杂度
    │                                              │
    │                                    ├─ 简单 → 直接回答用户
    │                                    │         （不创建 task，结束）
    │                                    │
    │                                    └─ 复杂 → 向用户说明："这个任务涉及多个步骤，建议制定一个执行计划。是否需要？"
    │                                              │
    │                                    用户确认 → 输出 [NEED_PLAN]
    │                                              │
    │                                              ▼
    │                                    llm_output 检测到标记
    │                                              │
    │                                    创建 task:{runId}.json（draft）
    │                                    注入 planning skill
    │                                              │
    │                                    Agent 制定 Plan
    │                                              │
    │                                    用户确认 → pending_approval → active
    │                                    （进入现有 v3.3.0 流程）
    │
    ├── task=draft ──→ 注入 planning skill → 制定/修改 Plan → 用户确认
    │
    └── task=active ─→ 注入 monitoring + 执行上下文 → 继续执行
```

## 关键设计决策

### 为什么增加"用户确认"环节？
- Agent 评估可能不准确（用户可能认为任务很简单）
- 尊重用户决策权，避免强制制定 Plan
- 符合核心原则：Plugin asks, Agent decides, **User confirms**

### 为什么用 `[NEED_PLAN]` 标记而不是工具调用？
- OpenClaw Hook 架构下，插件通过 `system context` 与 Agent 通信
- Agent 无法直接调用插件内部方法
- `[NEED_PLAN]` 标记是 Agent → 插件的最简通信方式
- 插件在 `llm_output` hook 中检测标记，完成创建，然后通过 `prependSystemContext` 追加 planning skill

### 简单任务的 tool call 是否记录？
- **不记录**。没有 task JSON 就没有存储容器
- 但 Session 全局索引（`working_memory:active_sessions`）仍正常工作
- 简单任务的 Session 创建/复用逻辑不变，只是不关联到任何 task

### 已创建的 draft task 如果 Agent 最终判断不需要 Plan？
- 理论上不应发生（Agent 在 assessment 阶段已判断）
- 若发生：task 保持 draft，`agent_end` 时 `archiveTask()` 检查 status，draft 不自动归档
- 可添加定期清理：删除超过 24h 的 draft task

## 与 v3.3.0 的兼容性

- **状态存储**：`task:{runId}` JSON 结构不变，只是创建时机延迟
- **Session 管理**：不受影响，简单任务仍可使用 Session（但不关联 task）
- **Personality 更新**：简单任务无 task → 无 event → `agent_end` 不触发 development skill
- **配置**：新增可选配置 `metacognition.assessment`（默认 true）
